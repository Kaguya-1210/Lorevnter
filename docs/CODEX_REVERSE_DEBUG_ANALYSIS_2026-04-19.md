# Lorevnter 逆向调试分析

日期：2026-04-19  
用途：提供给 Claude 4.6 Opus 参考的逆向调试结论  
说明：本文档只做“现象 -> 内部逻辑偏差”的反推，不提供修复代码，不给出具体改动方案。

---

## 现象一：测试模式按钮 UI 不一致

### 现象摘要

用户反馈调试模式中的“测试模式”相关按钮视觉表现明显偏离插件整体风格，怀疑项目中不止这一处存在类似问题。

### 可能的原因链条

#### 1. 最高概率：复用了类名，但没有真正复用样式体系

从源码看，`DebugTab.vue` 的测试模式区直接使用了 `st-row`、`st-label`、`st-select`、`st-btn`、`st-btn-test` 这些更像是 AI 配置页/设置页的样式类名，但 `DebugTab.vue` 自己的 `scoped` 样式中并没有完整定义这些类，只补了一个 `.st-btn-danger`。

这说明开发时很可能发生了下面的偏差：

- 开发者默认认为这些 `st-*` 类在整个插件中是“公共样式”
- 但实际上它们定义在别的单文件组件内部，且带有 `scoped`
- 因此在 `DebugTab` 中看起来“沿用了统一风格命名”，实际上样式并没有稳定生效

这类问题通常会直接导致：

- 同样的类名在不同页面表现不一致
- 某些控件退化成浏览器默认样式或半成品样式

#### 2. 较高概率：项目没有真正的共享 UI 基础层

从各个 tab 的结构看，按钮和表单行样式目前是按页面自行维护的：

- `st-*`：设置页/AI 配置页风格
- `wb-*`：世界书页风格
- `ct-*`：约束页风格
- `debug-*`：调试页风格

这意味着项目虽然视觉方向是统一的，但工程层并没有形成真正的“可复用设计系统”。  
开发新功能时，最容易发生的偏差就是：

- 想复用已有风格
- 但没有共享组件或共享样式文件
- 最终只能复制结构、借用命名、局部拼接

结果就是 UI 一致性依赖开发者记忆，而不是依赖系统约束。

#### 3. 中等概率：调试页在开发过程中被默认为“工程页”，没有按正式页面标准约束

`DebugTab.vue` 内部混用了多种表达方式：

- `debug-action-btn`
- `st-btn`
- 内联 `style`
- 单独定义的测试模式 banner

这种写法通常说明该区域是伴随调试功能逐步堆出来的，而不是从统一的页面模板或视觉规范出发构建的。  
一旦开发过程中把 Debug 区域视为“功能优先、样式次要”，风格漂移就会在这里最先出现。

### 需要重点关注的逻辑环节

- 组件样式隔离机制
  - 特别是 `scoped style` 下样式是否真的能跨组件复用
- 共享 UI 规范层
  - 是否存在真正的按钮、表单行、输入框、状态条基础层
- 调试页与正式页的开发分流
  - Debug 区是否被视为可以脱离主界面设计规范的例外区域
- 类名复用但样式未复用的地方
  - 这是本问题最可能重复出现的模式

### 对“其他地方是否也会重复出现”的反推

是，概率很高。

原因不是单个按钮写丑了，而是当前项目的 UI 一致性更像“约定俗成”，不是“系统强约束”。  
只要继续存在以下结构：

- 多个 tab 各自维护样式命名空间
- 没有抽离公共组件/公共样式
- 新功能开发时允许跨页面借用 class 名

那么类似问题会在其他 UI 组件中重复出现。

## 现象二：审核模式中变更展示逻辑混乱

### 现象摘要

用户明确指出审核模式应区分三种变更类型：

1. 新增条目
2. 条目内新增内容
3. 条目内修改内容

并且特别强调：

- “条目内修改内容”必须展示修改前和修改后的段落对比

当前实现显然没有正确区分这三类，尤其把条目内部变化的语义处理得过于粗糙。

### 可能的原因链条

#### 1. 最高概率：审核中间数据模型从一开始就不够细

当前审核项类型 `ReviewUpdate` 只有两种动作：

- `update`
- `create`

这意味着系统的内部模型最多只能表达：

- 这是已有条目的更新
- 这是一个新条目

但它不能表达：

- 这是对已有内容的追加
- 这是对已有内容的替换/改写

所以三种业务语义在类型层就已经被压扁成了两类。  
一旦中间模型不够细，后续 UI 再怎么渲染，也只能在错误粒度上做展示。

#### 2. 较高概率：分类判断写在“条目是否存在”这一层，而不是“文本变化类型”这一层

当前 `runUpdatePipeline()` 将 AI 返回结果转换成审核项时，判断依据本质上是：

- 如果能在世界书中找到同名条目 -> `update`
- 如果找不到 -> `create`

这类判断只能回答“目标条目是否已存在”，回答不了：

- 新内容只是追加了一段
- 还是修改了原有段落
- 还是同时发生了删改和补充

所以“条目内新增内容”和“条目内修改内容”在这一层大概率都被合并成同一种 `update`。

#### 3. 较高概率：渲染层只看动作类型，没有再做文本层细分

审核弹窗当前展示的是：

- 新增
- 修改

这种 UI 结构说明渲染层依赖的是粗粒度动作标签，而不是更细的文本差异语义。  
即使上游已经拿到了完整旧文本和新文本，只要没有独立的“差异语义识别”步骤，界面就仍会把所有内部变化都归类成“修改”。

#### 4. 中等概率：Diff 算法过于粗糙，导致“修改”在视觉上像“新增”

当前差异渲染逻辑是简化版逐行比较。  
这类逻辑在下面几种情况里很容易出错：

- 同一段内部改词
- 中间插入一句
- 原段落重写后结构仍然相似

当 diff 算法分辨率不够时，用户看到的不是“修改前 vs 修改后段落”，而是：

- 一段被删除
- 一段被新增

这会进一步加重“你把修改和新增搞混了”的体验。

### 需要重点关注的逻辑环节

- AI 结果标准化层
  - 模型输出转内部审核数据对象的地方
- 审核中间表示层
  - DTO / 类型系统是否有足够语义表达三种变化
- 变更分类判断层
  - 当前是否只按“条目存在性”而不是“文本变化性质”分类
- 审核界面条件渲染分支
  - UI 是否只支持 `update/create` 两分法
- 文本差异计算层
  - 是否采用了过粗的行级比较，导致视觉误判

### 对“修改内容为什么会被误判为新增或其它类型”的反推

最可能的根因不是某个 if 写反了，而是判断依据选错了层级。

当前结构更像是在回答：

- “这是不是一个已有条目的更新？”

而用户想要的是：

- “这个已有条目的更新，究竟是追加内容，还是修改原内容？”

两者不是同一个判断问题。  
如果系统只在“条目存在性”这一层做判断，那么“内部新增”和“内部修改”一定会混在一起。

### 从代码结构角度，这类判断逻辑通常写在哪里

通常会出现在下面两个位置：

- 数据处理管道中间层
  - 把 AI 输出转成审核对象的地方
- 审核 UI 适配层
  - 把审核对象转成具体渲染模型的地方

这两个位置最容易混淆的原因是：

- 上游拿到的是“旧文本 + 新文本”
- 开发者容易直接把它粗暴映射成“更新/新增”
- 没有再额外抽一层“文本差异语义识别”

所以混淆往往不是 UI 单层问题，而是中间表示层缺失导致的连锁后果。

## 现象三：正文提取规则配置无效

### 现象摘要

用户反馈 AI 配置中的“正文提取规则”无论是包含还是排除，实际效果都像没有配置一样。

### 可能的原因链条

#### 1. 最高概率：配置已保存、也被读取，但提取逻辑对真实文本结构的假设过于严格

从代码链路看：

- UI 通过 `v-model` 直接写入 settings
- pipeline 在 `getRecentChatMessages()` 中读取 `lore_context_include_tag` 和 `lore_context_exclude_tags`
- 非空时会调用 `extractContent()`

说明这条配置链理论上是接上的。

更像的问题是 `extractContent()` 的匹配策略太理想化：

- include 只支持严格的 `<tag>...</tag>`
- exclude 也只按严格标签包裹内容处理

如果真实消息并不满足这个格式，比如：

- 标签带属性
- 标签大小写不一致
- 宿主输出并非标准 XML
- 消息在进入此函数前后已被其它层处理

那么用户就会感知为“规则完全无效”。

#### 2. 较高概率：规则只作用于聊天消息正文，但用户观察的是更广义的“正文”

当前规则生效点在 `getRecentChatMessages()`，也就是“送给 AI 之前的聊天消息处理”。  
这意味着它不一定会影响用户能直接看到的所有内容来源。

如果用户观察的是：

- 调试面板的别处
- 宿主原始消息
- 世界书条目本身

那么即使规则在 AI 输入链路中执行了，也可能看起来像“没有生效”。

#### 3. 较高概率：规则执行后结果被后续过滤吞掉，用户感知不到变化

`extractContent()` 在 include 未匹配时会直接返回空文本。  
随后消息又会经过：

- 拼接角色前缀
- `trim`
- 长度过滤

如果提取结果过短，就会被整个过滤掉。  
这类情况下系统不是“完全没执行规则”，而是“执行后消息被隐式丢弃”，用户最终只会感觉“行为没有变化”。

#### 4. 中等概率：预览链路与正式执行链路不完全一致，造成用户误判

当前有一个“预览提取效果”功能，它取的是最近一条消息。  
正式执行时则是最近 N 条消息整体进入管线。  
如果用户主要通过预览判断规则效果，而正式流程中消息结构不同，那么就可能产生：

- 预览像不准
- 正式执行也看不出明显差异

最终统一感知成“配置无效”。

#### 5. 较低概率：保存失败或读取失败

从当前结构判断，这个概率反而不高。

原因：

- 配置是直接 `v-model` 到 settings store
- pipeline 也是直接从同一个 settings store 读取

这条链路没有明显的中间复制层或异步回填层，不像典型的“配置根本没存进去”的结构。

### 需要重点关注的逻辑环节

- 配置读取层
  - UI 写入值和 pipeline 读取值是否来自同一份状态
- 聊天消息预处理层
  - 规则是否只作用于 AI 输入链路，而不是用户理解中的全部正文
- 文本匹配执行层
  - 标签匹配是否过于理想化
- 后处理过滤层
  - 提取结果是否在后续被空值/短值过滤掉
- 预览链路与正式链路
  - 两者是否共享同一套输入假设

### 对“配置数据最可能在哪个环节丢失或被忽略”的反推

优先判断不是“丢失”，而是“被执行但效果不符合真实数据形态”。

按概率排序：

1. 执行层匹配不到真实消息结构
2. 执行后结果被后续过滤链吞掉
3. 用户观察位置与规则实际生效位置不一致
4. 读取失败
5. 保存失败

### 如果是执行环节被跳过，常见原因有哪些

在当前项目结构下，最常见的跳过原因可能是：

- include / exclude 最终读出来为空字符串
- 当前消息为空或不存在
- 文本不满足严格标签匹配格式
- 提取结果为空后被后续过滤
- 规则只挂在聊天消息链路，未覆盖用户实际关注的内容来源

## 总体逆向判断

这三个现象虽然表面上分属 UI、审核、正文提取三个模块，但底层问题很像，集中体现在三类偏差上：

### 1. 视觉规范依赖人工记忆，而不是系统约束

项目有统一视觉目标，但缺少真正的共享 UI 基础层。  
结果就是：

- 新功能开发时容易“借样式命名”
- 却没有真正继承一致的样式能力

### 2. 中间数据模型粒度不够

审核模式当前最像的问题不是“文案错误”，而是：

- 系统内部只建模了粗粒度动作
- 没有建模用户真正关心的变化语义

### 3. 配置链路看似接通，但执行层假设过于理想化

正文提取规则的问题更像是：

- 配置已经进入系统
- 但执行逻辑对真实输入格式的要求太严格
- 最终在用户层面表现为“规则无效”

---

## 适合交给 Claude 的一句话总结

当前三个 Bug 现象背后，不像是单点代码写错，更像是三个层面的结构性偏差：UI 没有真正共享的设计基础层，审核链路缺少足够细的中间语义模型，配置功能虽然接进了状态管理，但执行层对真实数据结构的假设过于理想化，导致用户看到的结果是“风格漂移、语义混淆、配置失效”。
