# Lorevnter 延期问题清单

> 记录日期：2026-04-19
> 来源：CODEX_PROJECT_AUDIT 审查报告 + GPT5.4 Codex 独立复核
> 状态：所有项均为**已确认、有意延期**，非遗漏

## 🔴 高优先级（暂缓但不降级）

### #1 变更分类器 `append/modify` 误判

**位置**：
- `src/Lorevnter/core/update-pipeline.ts:80-90`
- `src/Lorevnter/core/review-editor.ts:289-295`

**现状**：
当前 `append` 与 `modify` 的区分逻辑为 `trimNew.startsWith(trimOrig)`——新内容以旧内容开头就判定为追加，否则为修改。

**会出什么问题**：
- 内容重排、段落清洗、空白压缩、前缀保留但中段改写 → 误判
- 审核弹窗的 badge（新增/追加/修改）可能与实际操作不符
- 用户基于错误 badge 做出的审核决策可能不准确

**为什么现在不修**：
要修对就得上文本 diff 算法（LCS / 编辑距离），不是改一行能解决的。改半截（比如只加几个 heuristic）比不改更危险——会换一种方式误判，而且更难排查。

**什么时候修**：
核心管线跑稳（命中过滤、宏提示词、审核流程验证通过）后，作为**下一轮迭代首要目标**。

## 🟡 可延期（结构性技术债）

### #5 API Key 持久化边界不一致

**位置**：`src/Lorevnter/settings.ts:199-207`

**现状**：
预设导出时排除了 `lore_api_key`，但全量 settings 同步会把整个快照写入酒馆脚本变量——敏感字段和普通 UI 状态被同等持久化。

**为什么延期**：
要隔离就得把 settings 拆为「敏感层 + 普通层」，改持久化逻辑、导出逻辑、初始化逻辑——动面太广。而且酒馆本身是本地运行的桌面应用，API Key 存在脚本变量里的实际安全风险不高（不会暴露到网络）。

**触发条件**：如果未来支持云同步或设置导出分享，这条必须在此之前修。

---

### #6 备份 entries 导入校验偏弱

**位置**：`src/Lorevnter/core/backup-manager.ts:166-190`

**现状**：
备份记录的 `entries` 类型是 `any[]`，导入时只校验顶层字段存在，不校验条目内部结构。恢复时直接送进 `WorldbookAPI.save()`。

**为什么延期**：
要加校验就得先定义 `WorldbookEntry` 的标准 schema。但酒馆 API 返回的字段在不同版本可能不同，schema 定死了反而会导致合法备份被拒绝（比如老版本备份缺少新字段）。需要先摸清酒馆各版本的字段差异。

**触发条件**：如果出现用户导入损坏的 JSON 导致世界书污染的实际报告。

---

### #7 移动端响应式布局

**位置**：主窗口、Tab 导航、世界书列表、审核弹窗等全局

**现状**：
当前移动端靠 `overflow-x: auto` 和 `min-width` 兜底，不是真正的响应式重排。在手机端会出现信息截断、按钮挤压、需要横向滑动的问题。

**为什么延期**：
当前 UI 信息密度是按桌面设计的。要做移动端就得重新排信息优先级、折叠次要内容、调整触控区域——这是一个独立的 UI 重构任务，不应该混在功能修复里。

**触发条件**：移动端用户量增长或用户明确反馈移动端体验差。

---

### #9 包体积 365KB

**位置**：`webpack.config.ts` + 所有 Tab 组件同步导入

**现状**：
6 个 Tab 组件全部同步静态导入，构建产物触发 webpack `asset size limit` 告警。即使用户不打开调试页，也要承担 DebugTab 的首包成本。

**为什么延期**：
要拆包就得改 webpack 配置（`import()` 动态导入）、处理 chunk 命名和加载时序。对酒馆本地插件来说 365KB 不是致命问题——不会影响网络加载（本地文件），主要影响低性能设备首次解析速度。

**触发条件**：包体积超过 500KB，或用户反馈打开插件窗口卡顿。

---

### #10 原生 DOM + innerHTML 使用

**位置**：
- `src/Lorevnter/core/review-editor.ts:422-449`
- `src/Lorevnter/core/prompt-editor.ts:188-260`

**现状**：
审核弹窗和提示词编辑器用手写 DOM + `innerHTML` 构造，而非 Vue 组件。当前未发现可利用的 XSS 漏洞（部分输出经过 `escHtml` 处理），但会持续扩大遗漏转义、样式漂移、无障碍不足的风险面。

**为什么延期**：
重写为 Vue 组件意味着重构整个弹窗系统（事件绑定、状态管理、生命周期），是一个独立的技术迁移任务。当前实现功能上是正确的。

**触发条件**：新增弹窗功能时，优先用 Vue 组件实现；现有弹窗在下一轮 UI 重构迭代中逐步迁移。

---

### #11 设置深拷贝写回频率

**位置**：`src/Lorevnter/settings.ts:199-207`

**现状**：
每次设置变化都 `klona(settings.value)` 深拷贝后整体写回宿主变量。随着日志、缓存、预设、调试字段增加，写入成本会越来越高。

**为什么延期**：
当前规模下能正常工作，不会造成可感知的性能问题。优化方向是分层持久化（高频字段和低频字段分开写回），但这需要重构整个 settings 架构。

**触发条件**：设置项超过 50 个，或用户反馈设置页输入卡顿。

---

### #12 无自动化测试

**位置**：项目全局

**现状**：
`rg --files src | rg "\\.(spec|test)\\."` 无结果。核心链路（审核分类、备份恢复、正文提取、上下文刷新、测试模式回档）缺少自动回归保护。

**为什么延期**：
补测试不是写几个 `test()` 就行的——需要先搭建测试基础设施（Jest/Vitest 配置、酒馆 API mock、Vue Test Utils 集成），再为每条核心链路编写覆盖用例。这是一个完整的专项迭代。

**触发条件**：任何核心链路因回归 bug 被用户报告两次以上。
